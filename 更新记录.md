## [v0.16.8] - 2026-03-23

### 🔧 AI配置链路与后台生效修复

#### 更新原因

本次更新源于 AI 模型管理与后台 AI 分析链路存在两类相互叠加的问题，导致“界面配置正确但后台不生效”：

1. **AI管理页中的默认服务商只是界面默认值，没有真正同步到后台配置**
   - [`AiSettingsPage`](light_idea/lib/presentation/pages/ai_settings/ai_settings_page.dart:21) 中写死了默认聊天服务和默认 Embedding 服务。
   - 但后台任务实际通过 [`openAIClientProvider`](light_idea/lib/application/providers/app_providers.dart:34) 创建客户端，并读取 [`AIConfig.getApiBaseUrl()`](light_idea/lib/application/providers/app_providers.dart:35)。
   - 这导致 UI 中显示的是你配置的服务商，而后台分析链路却可能仍然回退到 [`AIConfig.apiBaseUrl`](light_idea/lib/config/ai_config.dart:19) 的默认值。

2. **原有默认 AIConfig 仍然指向 OpenAI 默认地址与模型，不符合当前国内可用配置策略**
   - 后台分析日志显示请求实际发送到了 [`https://api.openai.com/v1`](light_idea/lib/data/api/openai_client.dart:54)。
   - 这与当前项目要求“所有后台都应从用户配置或写死初始化配置生效”不一致。

3. **后台轮询过程中存在额外数据库异常，干扰了 AI 问题定位**
   - [`AIAnalysisRepositoryImpl.getByIdeaId()`](light_idea/lib/data/repositories/ai_analysis_repository_impl.dart:25) 中使用了 [`thenByIdDesc()`](light_idea/lib/data/repositories/ai_analysis_repository_impl.dart:29) 排序链。
   - 运行中触发 `IsarError: Unknown property "id"`，会导致轮询提前报错，影响对真实 AI 请求问题的判断。

4. **Android 主清单缺少正式网络权限声明**
   - 项目在 [`src/debug/AndroidManifest.xml`](light_idea/android/app/src/debug/AndroidManifest.xml:6) 与 [`src/profile/AndroidManifest.xml`](light_idea/android/app/src/profile/AndroidManifest.xml:6) 中有 [`INTERNET`](light_idea/android/app/src/debug/AndroidManifest.xml:6) 权限。
   - 但主清单 [`AndroidManifest.xml`](light_idea/android/app/src/main/AndroidManifest.xml:2) 原先缺失该权限，存在正式运行环境网络能力异常风险。

#### 详细修复内容

**1. 统一 AIConfig 默认生效值，确保后台默认不再回退到 OpenAI**

| 文件 | 变更类型 | 说明 |
| --- | --- | --- |
| `light_idea/lib/config/ai_config.dart` | 修改 | 将默认聊天地址、聊天模型、Embedding模型改为项目实际使用配置 |

具体调整：
- 将 [`AIConfig.apiBaseUrl`](light_idea/lib/config/ai_config.dart:19) 默认值改为 `https://api.zscc.in/v1`
- 将 [`AIConfig.defaultChatModel`](light_idea/lib/config/ai_config.dart:7) 改为 `gemini-3-flash`
- 将 [`AIConfig.defaultEmbeddingModel`](light_idea/lib/config/ai_config.dart:8) 改为 `Qwen/Qwen3-Embedding-8B`
- 将 [`AIConfig.getEmbeddingBaseUrl()`](light_idea/lib/config/ai_config.dart:97) 的回退值从 OpenAI embeddings 地址调整为统一复用 [`apiBaseUrl`](light_idea/lib/config/ai_config.dart:19)

**2. 新增 AI 服务商配置持久化能力，让 AI 管理页成为后台真实配置来源**

| 文件 | 变更类型 | 说明 |
| --- | --- | --- |
| `light_idea/lib/config/ai_config.dart` | 修改 | 新增 providers 配置的序列化持久化读写能力 |
| `light_idea/lib/presentation/pages/ai_settings/ai_settings_page.dart` | 修改 | 启动时加载 providers，变更后立即持久化并同步到后台配置 |

新增能力包括：
- [`AIConfig.saveProviders()`](light_idea/lib/config/ai_config.dart:129)
- [`AIConfig.getProviders()`](light_idea/lib/config/ai_config.dart:133)
- [`AIProvider.toJson()`](light_idea/lib/presentation/pages/ai_settings/ai_settings_page.dart:1313)
- [`AIProvider.fromJson()`](light_idea/lib/presentation/pages/ai_settings/ai_settings_page.dart:1300)

这意味着：
- AI 管理页中的 provider 列表不再只是页面临时状态。
- 配置会持久化到本地安全存储。
- 后续 App 重启后仍可保持当前服务商配置。

**3. 将默认 provider 真正同步写入后台 AIConfig，让后台任务统一读取你的配置**

| 文件 | 变更类型 | 说明 |
| --- | --- | --- |
| `light_idea/lib/presentation/pages/ai_settings/ai_settings_page.dart` | 修改 | 新增默认 provider 与后台 AIConfig 同步逻辑 |

核心新增逻辑：
- [`_buildDefaultProviders()`](light_idea/lib/presentation/pages/ai_settings/ai_settings_page.dart:51)：统一构建写死的初始化 provider
- [`_loadProviders()`](light_idea/lib/presentation/pages/ai_settings/ai_settings_page.dart:108)：优先加载持久化配置；若无配置则使用写死默认值初始化

同步内容包括：

- 聊天默认模型 → [`AIConfig.setChatModel()`](light_idea/lib/config/ai_config.dart:76)

- Embedding 默认密钥 → [`AIConfig.setEmbeddingApiKey()`](light_idea/lib/config/ai_config.dart:85)

并且在以下操作后都会立即同步：
- 保存服务商 [`_saveProvider()`](light_idea/lib/presentation/pages/ai_settings/ai_settings_page.dart:334)

**4. 修复后台 AI 分析轮询的 Isar 查询异常**

| 文件 | 变更类型 | 说明 |
| --- | --- | --- |
| `light_idea/lib/data/repositories/ai_analysis_repository_impl.dart` | 修改 | 去掉不兼容的二次排序，避免 `Unknown property "id"` |

修复点：
- 将 [`getByIdeaId()`](light_idea/lib/data/repositories/ai_analysis_repository_impl.dart:25) 中的排序链从：
  - [`sortByUpdatedAtDesc().thenByIdDesc()`](light_idea/lib/data/repositories/ai_analysis_repository_impl.dart:29)
- 调整为：
  - [`sortByUpdatedAtDesc()`](light_idea/lib/data/repositories/ai_analysis_repository_impl.dart:29)

修复效果：
- 后台轮询不再因为 Isar 排序异常中断。
- AI 失败日志能够更准确地反映真实网络或服务问题。

| 文件 | 变更类型 | 说明 |

新增内容：
- [`android.permission.INTERNET`](light_idea/android/app/src/main/AndroidManifest.xml:2)

#### 修复后的实际效果

本次更新后，AI 配置生效链路调整为：

1. AI 管理页初始化时，优先读取持久化 provider 配置。
2. 如果没有历史配置，则加载你写死的默认 provider。
3. 当前默认 provider 会立即同步到 [`AIConfig`](light_idea/lib/config/ai_config.dart:4)。
4. 后台分析、AI理解、Embedding、对话等服务通过 [`openAIClientProvider`](light_idea/lib/application/providers/app_providers.dart:34) 创建 [`OpenAIClient`](light_idea/lib/data/api/openai_client.dart:7) 时，会读取同步后的真实配置，而不是再回退到 OpenAI 官网地址。

也就是说：
- **AI 管理页显示的默认项 = 后台真正生效的默认项**
- **写死初始化配置不再只是展示，而是会进入全局配置并参与后台调用**
- **后台请求地址与模型将始终优先以你的配置为准**

- 任何获得源代码的人都可以使用该密钥

### 🐛 多条数据保存问题修复

#### 问题描述

用户反馈：当新建多条记录时，系统仅保存并显示第一条记录，后续创建的记录既未在界面显示，也未正确存储到数据库中。

**发现的根本原因**:

**问题: 轮询逻辑只支持单个任务**

在 `_pollAnalysisResultSilent` 方法中：

1. 保存第一条 → 开始轮询第一条
2. 保存第二条 → **取消第一条的轮询**，开始轮询第二条
3. 保存第三条 → **取消第二条的轮询**，开始轮询第三条

**结果**: 只有最后一条灵感会被正确轮询到结果！之前的轮询被取消了！

#### 修复内容

**1. 支持多个并发轮询** (`home_provider.dart`)

将单个 `_pollingTimer` 改为 `Map<int, Timer>` 存储多个轮询：

```dart
class HomeNotifier extends StateNotifier<HomeState> {
  final Ref _ref;
  final Map<int, Timer> _pollingTimers = {};  // 改为Map存储
  
  @override
  void dispose() {
    // 取消所有轮询
    for (final timer in _pollingTimers.values) {
      timer.cancel();
    }
    _pollingTimers.clear();
    super.dispose();
  }
  
  void _pollAnalysisResultSilent(int ideaId) {
    // 取消该ideaId的旧轮询（如果存在）
    _pollingTimers[ideaId]?.cancel();
    
    _pollingTimers[ideaId] = Timer.periodic(const Duration(seconds: 2), (timer) async {
      // ... 轮询逻辑 ...
      
      if (analysis.status == AnalysisStatus.completed) {
        timer.cancel();
        _pollingTimers.remove(ideaId);
        await loadIdeas();  // 刷新列表
      }
    });
  }
}
```

**2. AI更新时重新获取最新数据** (`ai_task_queue.dart`)

```dart

    categoryId: categoryId,

/// 后台静默轮询AI分析结果（不显示加载状态）
void _pollAnalysisResultSilent(int ideaId) {
  _pollingTimer?.cancel();
  
  int attemptCount = 0;
  const maxAttempts = 120; // 2分钟超时

  _pollingTimer = Timer.periodic(const Duration(seconds: 2), (timer) async {
    attemptCount++;
    
    if (attemptCount > maxAttempts) {
      timer.cancel();
      return;
    }

    try {
      final analysis = await analysisRepo.getByIdeaId(ideaId);
      
      if (analysis != null) {
        if (analysis.status == AnalysisStatus.completed) {
          // 分析完成，静默刷新列表
          timer.cancel();
          await loadIdeas();  // 自动刷新显示标签
        } else if (analysis.status == AnalysisStatus.failed) {
          timer.cancel();
        }
        // 分析中，继续轮询（静默）
      }
    } catch (e) {
      timer.cancel();
    }
  });
}
```

## \[v0.9.0] - 2026-03-14

### 🧭 路由系统重构

#### 问题描述

**原有问题**:
- 所有页面使用 `context.goNamed()` 进行导航
- `go()` 方法会替换当前路由而非压入导航栈
- 导致无法返回上一级页面（`context.pop()` 无效）

**影响范围**:
- 首页 → 灵感详情页：无法返回
- 首页 → AI中心：无法返回
- 首页 → 设置：无法返回
- 设置 → 回收站：无法返回
- 设置 → 帮助：无法返回

#### 解决方案

**采用方案二：ShellRoute 嵌套路由结构**

**核心改动**:

1. **路由结构重构** - 使用 ShellRoute 构建设置相关页面的嵌套结构
   ```dart
   ShellRoute(
     builder: (context, state, child) => _SettingsShell(child: child),
     routes: [
       GoRoute(path: '/settings', ...),
       GoRoute(path: '/recycle-bin', ...),
       GoRoute(path: '/help', ...),
     ],
   ),
   ```

2. **导航方式变更** - 将所有 `goNamed()` 改为 `pushNamed()`
   ```dart
   // 修改前
   context.goNamed(RouteNames.ideaDetail, pathParameters: {'id': id});
   
   // 修改后
   context.pushNamed(RouteNames.ideaDetail, pathParameters: {'id': id});
   ```

3. **扩展方法更新** - 更新 `GoRouterExtension` 提供便捷方法
   ```dart
   extension GoRouterExtension on BuildContext {
     void pushToIdeaDetail(String id);
     void pushToSettings();
     void pushToAiHub();
     // ...
   }
   ```

#### 修改的文件

| 文件 | 变更类型 | 说明 |
| --- | --- | --- |
| `lib/core/router/app_router.dart` | 重构 | 使用ShellRoute结构、添加错误页面、更新扩展方法 |
| `lib/presentation/pages/home/home_page.dart` | 修改 | goNamed → pushNamed |
| `lib/presentation/pages/settings/settings_page.dart` | 修改 | goNamed → pushNamed |

#### 路由层级结构

```
首页 (/) 
├── 灵感详情页 (/idea/:id)     [push进入，可返回]
├── AI灵感中心        [push进入，可返回]
├── AI设置    [push进入，可返回]
├── 数据管理 [push进入，可返回]
└── ShellRoute [设置相关页面]
    ├── 设置          [push进入，可返回]
    ├── 回收站   [push进入，可返回]
    └── 帮助            [push进入，可返回]
```

- `dorny/test-reporter@v1` 需要 `checks: write` 权限才能创建检查运行（Check Run）
- GitHub Actions 默认权限可能不足以创建检查运行
- 来自 fork 的 PR 没有写入权限

**解决方案**:

1. 添加必要的权限声明：
```yaml
permissions:
  checks: write
  contents: read
  pull-requests: write
```

2. 处理 fork PR 的情况：
```yaml
if: always() && (github.event_name != 'pull_request' || github.event.pull_request.head.repo.full_name == github.repository)
```

## \[v0.7.3] - 2026-03-14

### 🔧 Android构建配置修复

#### 修复内容

**isar_flutter_libs namespace 缺失问题** ✅

**错误信息**:

```
A problem occurred configuring project ':isar_flutter_libs'.
> Could not create an instance of type com.android.build.api.variant.impl.LibraryVariantBuilderImpl.
   > Namespace not specified. Specify a namespace in the module's build file.
```

**原因分析**:
- Android Gradle Plugin 8.6.0 强制要求所有模块声明 namespace
- `isar_flutter_libs` 是第三方包，其 `build.gradle` 未声明 namespace
- 这是 AGP 8.x 的破坏性变更

**解决方案**:

在项目根目录的 `build.gradle.kts` 中为所有子项目自动设置 namespace：

```kotlin
subprojects {
    afterEvaluate {
        if (project.hasProperty("android")) {
            project.extensions.configure<BaseExtension>("android") {
                compileSdkVersion(36)
                if (namespace == null) {
                    val manifestFile = file("${project.projectDir}/src/main/AndroidManifest.xml")
                    if (manifestFile.exists()) {
                        val manifestText = manifestFile.readText()
                        val packageRegex = """package\s*=\s*"([^"]+)"""".toRegex()
                        val packageName = packageRegex.find(manifestText)?.groupValues?.get(1)
                        if (packageName != null) {
                            namespace = packageName
                        }
                    }
                }
            }
        }
    }
}
```

**工作原理**:
1. 遍历所有子项目
2. 检查是否有 `android` 属性（即 Android 模块）
3. 如果 namespace 为空，从 `AndroidManifest.xml` 中提取 `package` 属性
4. 自动设置 namespace

**修改的文件**:

| 文件 | 变更类型 | 说明 |
| --- | --- | --- |
| `light_idea/android/build.gradle.kts` | 修改 | 添加自动设置 namespace 逻辑 |

#### 功能概述

| 文件 | 变更类型 | 说明 |

**3. 详情页与重新分析链路优化**

| 文件 | 变更类型 | 说明 |
| --- | --- | --- |
| `light_idea/lib/application/providers/idea_detail_provider.dart` | 修改 | 仅展示与当前快照匹配的分析结果；内容未变化时不重跑 AI；内容变化时清理旧任务与旧分析 |

| 文件 | 变更类型 | 说明 |
| --- | --- | --- |
| `light_idea/lib/application/task_queue/ai_task_queue.dart` | 修改 | 基础分析/关系分析写库前增加快照比对；仅在当前仍为 processing 时回写 completed |

#### 技术细节

1. `IdeaEntity` 在创建和 `copyWith()` 更新内容时自动计算 `contentHash`
2. `AIAnalysisEntity` 记录 `sourceContentHash + sourceIdeaUpdatedAt`，作为结果归属快照
3. AI 任务在分析完成后、写入分析结果前、写回 embedding/tag/category 前都会重新读取最新灵感并校验快照
4. 若快照不一致，则直接丢弃旧结果，并将灵感保持为 `AIStatus.pending`，等待新内容对应的任务继续处理
5. 详情页展示层也会做二次过滤，即使数据库中保留了历史分析，也不会展示给当前内容版本
