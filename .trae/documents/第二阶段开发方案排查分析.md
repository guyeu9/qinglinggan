## 三、优先级排序与方案分析

### 3.1 最高优先级：综合分析服务（P3）

根据产品设计说明书的核心理念，综合分析是 AI 智能关联的关键输出，直接影响用户对灵感关系的理解深度。

---

### 3.2 方案对比

#### 方案 A：完整实现（推荐）

**实施步骤**：
1. 扩展 `AIAnalysisEntity` 添加三个新字段
2. 扩展 `AIAnalysisModel` 同步添加字段
3. 创建 `ai_analysis_service.dart` 综合分析服务
4. 修改任务队列集成综合分析
5. 创建 `reanalyze_idea.dart` 重新分析用例
6. 更新灵感详情页展示综合分析结果

**优点**：
- 完整实现产品设计要求
- 数据结构完整，支持后续扩展
- 符合四层架构规范

**缺点**：
- 需要 Isar 数据库迁移
- 开发工作量较大（约 3-4 小时）

**风险评估**：
- 数据库迁移风险：低（Isar 支持 schema 自动迁移）
- API 成本风险：中（增加一次 AI 调用）

---

#### 方案 B：最小实现

**实施步骤**：
1. 仅创建 `ai_analysis_service.dart`
2. 在 `ai_chat_service.dart` 中添加综合分析方法
3. 不修改数据库结构，结果仅用于实时展示

**优点**：
- 无需数据库迁移
- 开发速度快（约 1-2 小时）

**缺点**：
- 综合分析结果无法持久化
- 每次查看都需要重新调用 AI
- 不符合设计文档要求

---

#### 方案 C：延迟实现

**实施步骤**：
1. 暂不开发综合分析功能
2. 先完成关联内容页面（P4）
3. 后续迭代再实现综合分析

**优点**：
- 避免数据库迁移风险
- 可以先验证关联功能的用户反馈

**缺点**：
- 功能不完整，用户体验受损
- 违背优先级排序原则

---

### 3.3 最佳方案：方案 A（完整实现）

**选择理由**：

1. **符合产品设计要求**：产品设计说明书明确要求 AI 输出"共同点、差异点、综合优化版本"
2. **数据持久化需求**：综合分析结果应保存，避免重复调用 AI
3. **架构一致性**：遵循四层架构，新增 Service 和扩展实体
4. **技术可行性**：Isar 数据库迁移成熟，风险可控

- 建议在 `pubspec.yaml` 中更新版本号

```dart
class AIAnalysisService {
  final OpenAIClient _openAIClient;
  final AppLogger _logger;

    required IdeaEntity currentIdea,
    required List<AssociationEntity> associations,

}

class AnalysisOutput {

  final List<String> differences;
  final String mergedIdea;
}
```
